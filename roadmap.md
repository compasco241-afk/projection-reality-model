# Development Roadmap

**Projection Reality Model — PRM**  
**A staged roadmap for conceptual development, documentation and publication**

## Purpose

This roadmap defines the current development path of the Projection Reality Model.

The purpose is to keep the project organized, transparent and intellectually responsible.

PRM is currently an early-stage conceptual research framework. The project should therefore develop step by step: first through clear vocabulary, then examples, then critique, then possible formalization and publication.

## Current Status

PRM currently has an initial public repository and a basic documentation structure.

The project already includes:

- general project description,
- Polish project description,
- core conceptual vocabulary,
- Reasoning Reviewer Gate concept,
- interpretation of time, energy and mass,
- interpretation of information and change,
- communication model,
- limitations and boundaries of the model.

This means that the project has moved from informal discussion to structured documentation.

## Stage 2: Internal Structure

**Status: in progress**

The goal of this stage is to make the internal structure of PRM more precise.

Planned documents:

- `roadmap.md`
- `working_relation.md`
- `examples.md`

### Key task: Working Relation

The most important concept requiring further development is **WR — Working Relation**.

WR should be described more precisely as:

- a stable relational unit,
- a structure that can preserve identity,
- a structure that can resist change,
- a structure that can participate in transformation,
- a structure that can project into observable form.

A separate document should explain what WR is, what it is not and how it can be used inside PRM.

### Key task: examples

The project needs examples showing how PRM can be applied.

Examples should include:

- communication,
- signal transmission,
- information loss,
- reasoning review,
- time as ordering of change,
- mass as relational resistance,
- energy as transformative capacity.

Examples should be clearly marked as conceptual, not as scientific proof.

## Stage 3: Critical Review

**Status: planned**

The goal of this stage is to test PRM against limitations, objections and counterexamples.

Planned work:

- identify weak points of the model,
- collect possible objections,
- create counterexamples,
- separate analogy from explanation,
- separate conceptual claims from physical claims,
- refine definitions.

Possible future document:

- `objections_and_responses.md`

This stage is important because a serious model must not only explain its own ideas. It must also expose itself to criticism.

## Stage 4: RRG Development

**Status: planned**

The Reasoning Reviewer Gate should be developed as the first practical application of PRM.

Planned work:

- create a detailed RRG checklist,
- define review categories,
- create example reviews,
- test RRG on real reasoning tasks,
- separate low-risk and high-risk review modes,
- define when an answer should be accepted, qualified, revised, rejected or verified.

Possible future documents:

- `rrg_checklist.md`
- `rrg_examples.md`

Main goal:

> Transform RRG from a conceptual idea into a practical reasoning-review workflow.

## Stage 5: Formalization Preparation

**Status: future**

The goal of this stage is not yet to create full mathematics, but to prepare PRM for possible formalization.

Planned work:

- clarify definitions,
- identify relations between concepts,
- create simple diagrams,
- define possible variables or structural markers,
- identify what could be measured,
- identify what cannot yet be measured,
- compare PRM concepts with existing scientific and philosophical frameworks.

Possible future documents:

- `formalization_notes.md`
- `concept_map.md`

## Stage 6: External Publication

**Status: future**

OpenScience, OSF, Zenodo or similar platforms should be used only after PRM has a sufficiently stable documentation base.

Before external publication, the project should include at least:

- general description,
- core concepts,
- limitations,
- roadmap,
- examples,
- detailed WR document,
- stable RRG description.

OpenScience should not be treated as the workshop stage.

GitHub is the workshop.  
OpenScience should be the presentation layer.

### Publication readiness criteria

PRM may be ready for OpenScience when:

1. the repository structure is stable,
2. the main concepts are clearly defined,
3. limitations are explicitly stated,
4. examples are available,
5. WR is explained in detail,
6. RRG has a practical description,
7. the project can be summarized clearly in one page.

## Stage 7: Long-Term Development

**Status: future**

Long-term development may include:

- Polish and English parallel documentation,
- diagrams,
- conceptual maps,
- comparison with information theory,
- comparison with systems theory,
- comparison with philosophy of science,
- AI reasoning-review experiments,
- possible article or preprint,
- versioned releases of the repository.

## Development Principles

PRM should follow these principles:

1. **Clarity before complexity**  
   Concepts should be understandable before they become more advanced.

2. **Honesty before ambition**  
   The model should not claim more than it can support.

3. **Structure before publication**  
   Public presentation should follow internal organization.

4. **Critique before certainty**  
   Objections and limits should be treated as part of development.

5. **Concept before formalization**  
   Mathematical or scientific formalization should come only after the conceptual structure is stable.

The PRM project should develop in stages:

1. foundation,
2. internal structure,
3. critique,
4. practical RRG development,
5. preparation for formalization,
6. external publication,
7. long-term refinement.

The current priority is not to publish too quickly.

The current priority is to make PRM clear, structured and defensible.
